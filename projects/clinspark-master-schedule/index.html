ClinSpark Master Schedule

Role: Solo Developer Timeline: 2025–Present Platform: Web (Cloudflare Pages + Render)

A production-deployed, full-stack scheduling web application built for clinical trial coordinators. It centralizes study milestone tracking, provides a Gantt chart view across all active studies, enables real-time multi-user presence awareness, and enforces role-based access — replacing a fragmented collection of spreadsheets with a single, purpose-built tool.

Live Demo Private Repository

The Problem

Clinical trial E-Source Builders manage dozens of overlapping studies simultaneously, each with a sequence of critical milestones (screening, admission, dosing, UAT, build test, etc.) that must be tracked for multiple cohorts and study events. The existing process relied on shared Excel spreadsheets:

  • No live visibility into who was editing — concurrent edits caused data loss.
  • Milestone dates were scattered across separate tabs and files, making cross-study scheduling impossible at a glance.
  • No audit trail: changes were anonymous and irreversible.
  • Generating a formatted schedule for distribution required hours of manual reformatting.
  • Status and priority tracking (blocked, pending, confirmed) were informal and inconsistently maintained.

Solution Overview

Study & Milestone Management

  • Create and edit studies with protocol number, sponsor, PI, and phase
  • 7 milestone rows per study: Screening, Admission, Dosing, Change Log, UAT, Build Test, Procedure Log
  • Per-milestone: date, status, priority, assignee, and notes fields
  • Archive and restore studies to keep the active view focused
  • Inline editing with optimistic UI and conflict detection

Gantt Chart View

  • Timeline visualization spanning all active studies and their milestone dates
  • Color-coded by milestone type for at-a-glance cross-study scheduling
  • Zoomable and scrollable date axis
  • Tooltip details on hover: study name, milestone, date, status, assignee
  • Filterable by study, milestone type, and date range

Real-Time Presence

  • Heartbeat polling tracks which users are currently active
  • Per-study edit locks prevent simultaneous overwrites
  • Active users bar shows who is online across the entire platform
  • Session-level lock claiming with automatic expiry on disconnect
  • Presence state updates broadcast on heartbeat intervals

Security & Access Control

  • JWT authentication stored in HttpOnly cookies — no token exposure in JavaScript
  • JTI-based session revocation: logout immediately invalidates the session server-side
  • Refresh token rotation on every renewal cycle
  • Brute-force rate limiting on login (per IP, per username, per IP+username)
  • Role-based access: Admin can manage users; Staff can view and edit studies
  • Full security headers on every response (CSP-ready, X-Frame-Options, HSTS)

Export & Audit

  • One-click Excel export of the full schedule, formatted for distribution
  • Complete audit trail: every create, edit, and delete is logged with user and timestamp
  • Audit log viewer with date-range filtering and export
  • Change-tracker banner alerts users when another user has modified the schedule

User Experience

  • Favorites system: pin frequently accessed studies to the top
  • Persistent dark/light mode toggle per user
  • Responsive layout that works on tablets and wide monitors
  • Debounced inline save with visual feedback on each cell
  • Column sort and archive filter on the main table view

Architecture

The application follows a clean separation between a React SPA frontend and a Python FastAPI backend, communicating over a REST API with JSON. PostgreSQL handles all persistent state. The frontend is served as a static site from Cloudflare Pages; the backend runs on Render's managed container platform.

  • Frontend: React 18, TypeScript, TailwindCSS, Zustand (state), Axios (HTTP), Recharts (Gantt)
  • Backend: Python, FastAPI, psycopg (async PostgreSQL driver), python-jose (JWT), PBKDF2-SHA256 (passwords)
  • Database: PostgreSQL with versioned migrations applied at startup
  • Deployment: Cloudflare Pages (frontend) + Render (backend API)
  • Cross-origin auth: SameSite=None; Secure cookies with CORS credentials enabled

Challenges & Solutions

  • Cross-origin cookie authentication: The frontend (Cloudflare Pages) and backend (Render) sit on different top-level domains, which blocks SameSite=Lax cookies in cross-site XHR. The solution required SameSite=None; Secure=true cookies paired with CORS credentials — with clear environment-variable-driven configuration so local development still uses permissive defaults.
  • Edit conflict prevention: Multiple coordinators working simultaneously could overwrite each other's changes. I implemented a server-side session-locking model: the first user to open a study's edit row claims a lock (stored in the database with an expiry); subsequent users see the lock owner and cannot edit until it's released or times out.
  • Session revocation at logout: Standard JWTs are stateless and can't be revoked before expiry. I introduced a sessions table keyed by JTI. Every authenticated request checks the JTI against this table; logout deletes the row, immediately invalidating the token without waiting for expiry.
  • Versioned database migrations: Schema changes needed to apply automatically on each deploy without requiring manual intervention. I wrote a lightweight migration runner that compares the current schema version against ordered SQL files and applies only the delta.
  • Change-awareness across users: When one user saves changes, other users should be notified without a full page reload. A polling endpoint (/studies/changes) returns a counter and "last changed by" field; the frontend polls every 30 seconds and shows a dismissible banner when a change is detected.

Outcomes & Impact

  • Replaced disconnected Excel spreadsheets with a single source of truth accessible to the entire team from any device.
  • Edit conflicts eliminated through server-side locking — no more lost changes.
  • Schedule exports that previously took 30–60 minutes of manual formatting now take one click.
  • Full audit trail allows supervisors to review who changed what and when — a compliance requirement in clinical research environments.
  • Deployed and in active use by the clinical operations team.

Lessons Learned

  • Cross-origin cookie authentication is deceptively tricky in production. Browser SameSite policies, Secure requirements, and CORS headers all interact — documenting the expected values for each environment (local, staging, production) upfront would have saved debugging time.
  • Stateless JWTs and session revocation are fundamentally at odds. The JTI table pattern works well, but it's important to set realistic expectations about the overhead of one DB lookup per authenticated request versus the security guarantee it provides.
  • Optimistic UI updates feel fast but require careful rollback logic. Every inline save needs a clear "saving…", "saved", and "failed" state so users trust the data persisted correctly.
  • Automated database migrations are worth the investment early. Adding them after tables have data is significantly harder than designing the migration system from the first table.

Future Improvements

  • WebSocket-based live updates to replace the polling change-detection system.
  • Study template system: clone a previous study's milestone structure as a starting point for new studies.
  • Email/notification integration: alert assignees when a milestone is updated or a deadline is approaching.
  • Mobile-optimized view for quick read-only lookups on phones.
  • CSV import to bulk-create or update studies from existing spreadsheet data.
